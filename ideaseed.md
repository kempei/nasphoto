# 写真管理システム

本文書は写真管理システムについてのアイデアを述べているものである。

写真を保存するのはオンプレミスのomv (Open Media Vault) サーバーで、現在2TBを超えたところ。今後も増えていくだろう。

写真はiPhoneやミラーレス一眼レフカメラで撮影され、Mac miniの写真.appにまず保存されている。

Macから omv に対してできるだけ即座に、安定して写真と動画のバックアップを取り続ける仕組みが必要。どこまでバックアップされているかについては、全件検索する必要はなく、写真.appに取り込んだ日時ベースでの判定で十分。Mac上で動作し、バックグラウンド処理を行うネイティブアプリが良い。

写真や動画に関するできるだけ多くのメタデータについて、もともとの写真に付随するメタデータはもちろん、「含まれているオブジェクト」「シーン」などを Mac 上のローカルGPUや Apple の SDK を駆使して解析する。さらに紙の写真をスキャンした場合に欠損しているメタデータについては、ユーザとの対話や前後の写真の情報からの類推など補完する。得られたメタデータは、Supabase上の無料枠にて稼働するクラウドDBに保存する。

iPhoneで動作するネイティブアプリにてビューアー機能を実装する。写真や動画のサムネイルを適切にキャッシュし、ストレスなく閲覧ができる。また、キーワードや自然言語による検索ができ、クラウドDBと連携して高速に検索結果の表示と閲覧ができる。自然言語からDBクエリへの変換は適切な前提情報を渡した上でClaude Haikuなどのコストパフォーマンスの良いLLMを用いる。

omv 側がNASのインターフェースしか使えない状況ではパフォーマンスが期待できないため、UDPや非同期転送など本アプリの目的に特化したパフォーマンス上の機能を持つMacやiPhoneとやり取りをする軽量なAPIサーバーを立てると良いと考える。同時接続数はほぼ1で、稀に2程度という特性を活かした実装にできる。

アプリやAPIの認証はauth0を使いシンプルにする。Supabase のAPIキーなどの鍵情報はAWSのパラメータストアを用いる。AWSへのアクセス情報は、Macであれば環境上デフォルトのアクセスキーを用いる。オンプレミスについては/tmp配下に鍵が連携される仕組みがあるのでそれを利用する。iPhoneについてはAWSには直接アクセスせず、オンプレミス上のAPIサーバーに対して auth0 認証後にトークンでアクセスし、ロールによる認可を APIサーバー側で実施する方式とする。
